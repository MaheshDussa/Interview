# Azure Resources Setup

This document describes the Azure resources needed to run the TaskMgmt application across `dev`, `qa`, `stage`, and `prod`, what each resource is used for, and how those resources connect to the frontend application in this repository.

## Architecture summary

This repository contains the Angular frontend for TaskMgmt.

The frontend depends on these external services:

- a frontend hosting service
- a backend Task API
- Microsoft Entra ID for authentication when `auth.provider` is `azureAd`
- Azure Application Insights for client-side telemetry when `appInsights.enabled` is `true`

## Important current behavior

The current deployment script in [deploy-angular.ps1](deploy-angular.ps1) deploys the `dist/task-mgmt/browser` output to Azure App Service as a static Angular SPA.

That means:

- the current production-style deployment path is SPA hosting
- the SSR Node server in [server.ts](server.ts) is present in the repo, but it is not what the current deployment script publishes
- if you want true Angular SSR in Azure, you would need a separate Node-based deployment approach

## Resource checklist per environment

Create the following for each environment that you use:

| Resource | Dev | QA | Stage | Prod | Purpose |
|---|---|---|---|---|---|
| Resource Group | Recommended | Required | Required | Required | Logical container for all resources |
| Frontend App Service | Optional for local-only dev | Yes | Yes | Yes | Hosts the Angular frontend |
| App Service Plan | Shared or dedicated | Yes | Yes | Yes | Compute plan for the frontend App Service |
| Backend API App Service | If API is hosted in Azure | Yes | Yes | Yes | Hosts the task management API |
| API App Service Plan | Shared or dedicated | Yes | Yes | Yes | Compute plan for the backend API |
| Entra SPA App Registration | Yes | Yes | Yes | Yes | Auth for the Angular frontend |
| Entra API App Registration | If using Azure AD auth | Yes | Yes | Yes | Exposes delegated API scope |
| Application Insights | Optional | Recommended | Recommended | Required | Client telemetry for the frontend |
| Log Analytics Workspace | Optional | Recommended | Recommended | Recommended | Centralized monitoring if needed |
| Key Vault | Optional | Recommended | Recommended | Recommended | Store app secrets and connection strings |
| Custom Domain + TLS | Optional | Optional | Recommended | Recommended | Public access with trusted URL |

## Recommended Azure layout

Use one resource group per environment.

Suggested naming pattern:

- `rg-taskmgmt-dev`
- `rg-taskmgmt-qa`
- `rg-taskmgmt-stage`
- `rg-taskmgmt-prod`

Suggested frontend resource names:

- App Service plan: `asp-taskmgmt-fe-<env>`
- Frontend App Service: `app-taskmgmt-fe-<env>`
- Application Insights: `appi-taskmgmt-fe-<env>`

Suggested backend resource names:

- App Service plan: `asp-taskmgmt-api-<env>`
- Backend App Service: `app-taskmgmt-api-<env>`
- Application Insights: `appi-taskmgmt-api-<env>`

## Resources to create

### 1. Resource Group

Create one resource group for each environment.

Purpose:

- keeps frontend, backend, monitoring, and identity-related configuration grouped together by environment
- makes deployment, cost tracking, and cleanup simpler

Example:

```powershell
az group create --name rg-taskmgmt-qa --location centralindia
```

### 2. Frontend App Service

Create an Azure App Service to host the built Angular frontend.

Purpose:

- serves the Angular SPA files generated by `ng build`
- exposes the public frontend URL used in Azure AD redirect URIs

Recommended settings:

- OS: Windows is acceptable for the current `web.config` rewrite setup
- Runtime: Node is not required if you continue publishing only the browser build
- HTTPS only: enabled
- Always On: recommended for non-dev environments

How this repo uses it:

- [deploy-angular.ps1](deploy-angular.ps1) zips `dist/task-mgmt/browser`
- the zip is deployed with `az webapp deploy`
- [web.config](web.config) rewrites client-side routes back to `index.html`

Example:

```powershell
az appservice plan create --name asp-taskmgmt-fe-qa --resource-group rg-taskmgmt-qa --sku B1 --is-linux false
az webapp create --name app-taskmgmt-fe-qa --resource-group rg-taskmgmt-qa --plan asp-taskmgmt-fe-qa --runtime "NODE|20-lts"
```

### 3. Backend API App Service

Create or reuse a backend API hosting resource.

Purpose:

- serves the Task API endpoints used by the frontend
- handles `/api/Tasks` and, when local auth mode is used, `/api/Auth/login`

How this frontend uses it:

- every environment file sets `apiBaseUrl`
- the Angular app calls `${apiBaseUrl}/api/Tasks`
- local login mode calls `${apiBaseUrl}/api/Auth/login`
- Azure AD mode attaches a bearer token to requests going to `apiBaseUrl`

Backend requirements:

- enable CORS for each frontend origin
- if `auth.provider` is `local`, keep the existing login endpoint working
- if `auth.provider` is `azureAd`, validate Microsoft Entra JWT access tokens

### 4. Microsoft Entra ID App Registrations

If Azure AD authentication is used, create two app registrations per environment or define a clear shared-tenant strategy.

#### SPA app registration

Purpose:

- authenticates users into the Angular frontend
- issues tokens the SPA can use to call the backend API

Required configuration:

1. Azure Portal -> Microsoft Entra ID -> App registrations -> New registration.
2. Set platform type to `Single-page application`.
3. Add redirect URIs for the environment.
4. Add post-logout redirect URIs.
5. Add API permission for the backend API scope.
6. Grant admin consent if required.

Typical redirect URIs:

- Dev: `http://localhost:4200/login`
- QA: `https://your-qa-app-url/login`
- Stage: `https://your-stage-app-url/login`
- Prod: `https://your-prod-app-url/login`

Values used by this repo:

- `azureAd.clientId`
- `azureAd.tenantId`
- `azureAd.redirectUri`
- `azureAd.postLogoutRedirectUri`

#### API app registration

Purpose:

- represents the protected backend API in Entra ID
- exposes a delegated scope that the SPA requests

Required configuration:

1. Open the backend API app registration.
2. Go to `Expose an API`.
3. Set the Application ID URI.
4. Create a scope such as `access_as_user`.
5. Add that scope to the SPA registration as a delegated permission.

Value used by this repo:

- `azureAd.apiScopes`

Example:

```text
api://<api-client-id>/access_as_user
```

### 5. Application Insights

Create one Application Insights resource per environment.

Purpose:

- collects frontend telemetry from this Angular app
- captures startup failures, filtered unhandled app errors, route page views, and explicit app/API failure logs

How this repo uses it:

- telemetry is controlled by `appInsights.enabled`
- the frontend uses `appInsights.connectionString`
- the cloud role name is set through `appInsights.cloudRole`

Current logging design in this repo:

- automatic AJAX tracking is disabled
- automatic fetch tracking is disabled
- automatic broad exception tracking is disabled
- explicit logs from app code are sent
- global errors are filtered to reduce library and framework noise

Recommended environment values:

- Dev: disabled by default or enabled with low-volume testing
- QA: enabled
- Stage: enabled
- Prod: enabled

### 6. Key Vault

Key Vault is not directly wired into this frontend repo, but it is recommended for managing environment-specific secrets and operational settings.

Use it for:

- backend API secrets
- deployment credentials
- App Insights connection strings for deployment pipelines
- any server-side configuration not meant to live in source control

Important:

- do not put sensitive secrets directly into frontend environment files if the value must remain secret
- frontend environment files are bundled into client-side JavaScript and are visible to users

Safe frontend configuration examples:

- public frontend URL
- public API base URL
- Entra tenant ID
- Entra SPA client ID
- App Insights connection string if your team accepts it for client telemetry

Not safe to expose in frontend code:

- client secrets
- API keys intended to be private
- database connection strings
- Key Vault secrets

## Deployment flow for the frontend

### Current deployment method in this repo

1. Build the frontend.
2. Package the `dist/task-mgmt/browser` folder.
3. Deploy the zip to the frontend App Service.

The provided script is [deploy-angular.ps1](deploy-angular.ps1).

Typical flow:

```powershell
ng build --configuration production
.\deploy-angular.ps1
```

Before using the script, update these values in [deploy-angular.ps1](deploy-angular.ps1):

- resource group name
- App Service name
- local source folder path
- local zip file path

## End-to-end configuration checklist

For each environment, verify all of the following:

1. Frontend App Service URL is created and reachable.
2. Backend API URL is created and reachable.
3. Frontend environment file points to the correct `apiBaseUrl`.
4. Backend CORS allows the frontend origin.
5. If using `local` auth, `/api/Auth/login` works from that environment.
6. If using `azureAd`, the SPA app registration is configured.
7. If using `azureAd`, the API app registration exposes the delegated scope.
8. If using `azureAd`, the backend validates the token audience and tenant.
9. If using App Insights, the connection string is set and telemetry is enabled.
10. The Angular build for that environment completes successfully.

## Suggested rollout order

1. Create resource groups.
2. Create backend API hosting.
3. Create frontend App Service hosting.
4. Create Entra app registrations.
5. Configure backend authentication and CORS.
6. Create Application Insights resources.
7. Fill in the environment files.
8. Deploy QA.
9. Validate sign-in, task CRUD, and telemetry.
10. Promote the same model to stage and prod.

## What this repo does not provision

This repository does not currently include IaC for creating Azure resources.

If you want resource creation to be repeatable, the next step should be one of these:

1. add a Bicep template for all frontend resources
2. add Terraform for frontend, auth, and monitoring resources
3. add an Azure DevOps or GitHub Actions pipeline that builds and deploys environment-specific bundles
